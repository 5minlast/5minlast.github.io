# [Save to youth] AIDevelopment Guide
## 모바일 웹 서비스 구축용 최종 지시서

---

# 1. 프로젝트 정의

## 프로젝트명
**Save to youth: 청소년 안전 종합 지역 정보 서비스**

## 프로젝트 목적
이 프로젝트는 **청소년, 학부모, 선생님**이 접속하면 원하는 지역의 **청소년 안전, 마음건강, 지원기관, 도움 연결 정보**를 한 번에 확인할 수 있는 **모바일 중심 HTML 웹 서비스**를 만드는 것이다.

기존에 Streamlit으로 만든 대시보드가 이미 존재하며, 이를 새로 버리지 않고 **핵심 구조·데이터·시각화 자산을 최대한 재활용**하여 **서비스형 모바일 웹**으로 전환한다.

이 서비스는 단순 조회형 대시보드가 아니라 다음 세 가지를 동시에 충족해야 한다.

1. **지역별 청소년 안전망 탐색**
2. **지역별 위험/지원 지표 비교와 해석**
3. **실제 도움 기관으로의 연결**
4. **안전한 이동 경로 결정**

개발 및 구현은 **안티그라비티**에서 진행한다.

---

# 2. 이 문서의 목적

이 문서는 AI가 아래 내용을 **오해 없이 바로 실행 가능한 수준**으로 이해하도록 작성한다.

- 서비스 목표
- 사용자
- 기능 범위
- 기존 Streamlit 산출물 재활용 방식
- 데이터 구조
- 자동화/API 운영 원칙
- 모바일 UX 원칙
- 시각화/해석 규칙
- 품질 기준
- 산출물 요구사항

이 문서는 **기획 설명서**가 아니라 **구현 지시서**다.  
따라서 AI는 문서를 읽고 바로 다음 작업으로 들어갈 수 있어야 한다.

# 3. 핵심 사용자

## 3.1 청소년
청소년은 다음 정보를 원한다.
- 내가 사는 지역에서 도움받을 수 있는 곳
- 상담 가능한 기관
- 쉼터, 청소년센터, 수련시설
- 우리 지역이 다른 지역과 비교해 어떤 상황인지
- 정신건강/안전 관련 지표가 어떤 의미인지
- 위기 시 어디에 연락하면 되는지
- 이동 경로를 설정하면 위험한 지역(유해업소, 범죄 다발지역, 교통사고 등)을 피해서 이동 경로를 설정할 수 있음
- 제보를 통한 위험지역 설정 및 회피 경로 확인

## 3.2 학부모
학부모는 다음 정보를 원한다.
- 우리 지역의 청소년 안전 인프라
- 위험 신호를 과장 없이 이해할 수 있는 자료
- 위기 시 바로 연락 가능한 기관 정보
- 공식 출처 기반의 신뢰 가능한 데이터
- 자녀를 위한 지역 지원 자원
- 이동 경로를 설정하면 위험한 지역(유해업소, 범죄 다발지역, 교통사고 등)을 피해서 이동 경로를 설정할 수 있음
- 제보를 통한 위험지역 설정 및 회피 경로 확인

## 3.3 선생님
선생님은 다음 정보를 원한다.
- 지역 단위 청소년 안전망 현황
- 학생 지원에 활용 가능한 기관 정보
- 지역 비교 통계
- 설명 가능한 시각화와 해석 문구
- 학교 현장에서 참고 가능한 데이터 기반 자료
- 이동 경로를 설정하면 위험한 지역(유해업소, 범죄 다발지역, 교통사고 등)을 피해서 이동 경로를 설정할 수 있음
- 제보를 통한 위험지역 설정 및 회피 경로 확인

# 6. 기존 Streamlit 자산 재활용 원칙

## 이미 존재하는 Streamlit 구현 요소
다음 자산은 최대한 재사용한다.

- 다크/라이트 모드(가독성 확보 필요 과제)
- KPI 카드 UI
- Plotly 상호작용 차트
- PyDeck 기반 지도
- 대한민국 마스킹(Inverted Polygon)
- 레이어 on/off
- 2D/3D 지도 시각화
- 지역 비교 차트
- 해석 가이드 문구
- 기존 `app.py`
- 기존 `requirements.txt`
- 기존 가공 데이터
- 기존 스타일 로직

## 전환 원칙
기존 Streamlit 앱은 “위험 분석형 대시보드” 성격이 강하다.  
HTML 모바일 웹 전환 시에는 이를 아래 방향으로 바꾼다.

### 기존 성격
- 대시보드 중심
- 분석 중심
- 위험 시각화 중심

### 새 성격
- 서비스형 정보 제공
- 탐색 + 비교 + 연결 중심
- 실제 기관 접근성 강화
- 모바일 우선
- 직관적인 정보 제공
- 공공api를 이용한 정보의 자동화
- 위험지역 제보 및 회피 경로 확인
- 제보를 통한 위험지역 설정 및 회피 경로 확인
- 제보를 통한 직접적 능동성 확보

즉, **기존 구조(Streamlit)의 기능을 최대한 재활용하여, 사용자 경험과 정보 구조를 서비스형으로 전환**한다.

# 7. 핵심 서비스 구조

## 7.1 홈
### 목적
- 서비스 소개
- 핵심 KPI 확인
- 지역 선택 유도
- 사용자 유형별 진입 제공
- 최신 데이터 기준일 제시

### 필수 요소
- 서비스 제목
- 서비스 설명
- 지역 선택 진입 버튼
- 핵심 지표 3개
- 데이터 최신 기준일
- 청소년 / 학부모 / 선생님별 바로가기

---

## 7.2 지역 선택
### 목적
사용자가 원하는 지역을 선택하면 이후 지도/차트/기관 정보가 모두 해당 지역 기준으로 바뀌게 한다.

### 필수 요소
- 시도 선택
- 시군구 선택(가능한 경우)
- 내 위치 기반 제안(선택사항)
- 최근 선택 지역 저장(선택사항)

---

## 7.3 안전망 지도
### 목적
청소년 관련 지원기관을 지도에서 확인하고 실제로 도움으로 연결되게 한다.

### 필수 기능
- 2D 지도
- 3D 지도
- 지도 모드 전환 버튼
- 레이어 on/off
- 지역 필터 반영
- 기관 유형 필터 반영
- 상세 기관 카드
- 모바일 터치 최적화

### 표시 대상
- 청소년 수련시설
- 청소년상담복지센터
- 청소년쉼터
- 필요 시 기타 청소년 지원기관

### 클릭 시 보여줄 정보
- 기관명
- 기관 유형
- 주소
- 전화번호
- 홈페이지
- 운영기관
- 운영 상태
- 간단한 이용 안내

### 지도 원칙
- 2D 지도는 탐색 편의 중심
- 3D 지도는 시각적 강조 중심
- 사용자가 자유롭게 2D/3D 전환 가능해야 함
- 색상만으로 의미를 전달하지 말고 텍스트/아이콘 병행

---

## 7.4 지역 비교 시각화
### 목적
선택 지역과 타 지역을 비교하여 안전망 수준, 위험 신호, 지원 수요를 이해하도록 돕는다.

### 가능한 시각화 예시
- 지역별 안전망 시설 수
- 청소년 10만 명당 안전망 시설 수
- 정신건강 지표
- 상담지원 수요/지원 건수
- 추이 그래프
- 시설 유형 비중 도넛 차트

### 시각화 공통 요구사항
모든 차트는 다음을 반드시 포함한다.
- 제목
- 단위
- 기준연도
- 출처
- 해석 가이드
- 데이터 한계 또는 주의문

### 해석 규칙
모든 시각화에는 최소 1개 이상의 해석 문구를 넣는다.

예:
- “값이 높을수록 안전망이 상대적으로 잘 갖춰진 것으로 해석할 수 있습니다.”
- “값이 높을수록 주의가 필요한 신호일 수 있습니다.”
- “지역 비교를 위한 참고 자료이며 단순 인과관계를 의미하지 않습니다.”
- “조사 방식과 기준연도 차이를 고려해 해석해야 합니다.”

### 금지 사항
- 인과관계를 단정하는 문구
- 공포를 조장하는 표현
- 낙인감을 줄 수 있는 표현
- 근거 없는 긍정/부정 해석

---

## 7.5 도움받기
### 목적
사용자가 데이터 조회에서 끝나지 않고 실제 기관/상담 지원으로 이어지게 한다.

### 필수 요소
- 대표 상담 연락처
- 지역 상담복지센터 목록
- 청소년쉼터 목록
- 긴급 도움 안내
- 전화하기 버튼
- 홈페이지 바로가기 버튼
- 지도에서 보기 버튼
- 상황별 도움 안내

### 상황별 도움 안내 예시
- 우울/불안
- 학교 문제
- 폭력/괴롭힘
- 가출/가정 문제
- 진로/생활 고민

---

## 7.6 데이터 설명 / 팩트체크
### 목적
데이터 신뢰성을 확보하고 사용자가 출처를 확인할 수 있게 한다.

### 필수 요소
- 지표 정의
- 산출 기준
- 기준연도
- 제공기관
- 원문 링크
- 다운로드 링크(가능한 경우)
- 데이터 한계
- 해석 유의사항

### 원칙
- 모든 수치에는 출처가 있어야 한다
- 사용자는 “이 수치가 어디서 왔는지” 확인 가능해야 한다
- “알 수 있는 것 / 알 수 없는 것”을 분리해서 보여준다

# 10. 자동화 및 API 키 운영

## 목표
API 키를 사용해 데이터 최신 여부를 **자동으로 점검하고 반영 가능한 구조**를 만든다.

## 자동화가 해야 하는 일
- 최신 데이터 유무 확인
- 갱신 주기별 체크
- 원본 데이터 수집
- 전처리 실행
- 품질 검증
- 가공 파일 생성
- 배포 반영
- 실패 시 알림
- 변경 로그 기록

## 자동 점검 항목
- 출처 링크 유효성
- 기준연도 변경 여부
- 기관 정보 변경 여부
- 좌표 이상치
- 중복 데이터
- 누락 데이터
- 스키마 오류

## API 키 관리 원칙
- 코드에 하드코딩 금지
- 환경 변수 또는 시크릿 관리 시스템 사용
- 권한 최소화
- 사용량/실패 로그 기록
- 키 노출 방지

## 자동화 파이프라인 예시
1. 외부 API/원본 확인
2. 원본 다운로드/동기화
3. 전처리
4. 검증
5. 결과 CSV 생성
6. 사이트 반영
7. 운영 로그 저장

# 11. 모바일 우선 설계 원칙

## 핵심 원칙
- 모바일에서 먼저 완성도 있게 보여야 한다
- 한 손 사용 가능해야 한다
- 카드형 정보 구조를 사용한다
- CTA 버튼은 크게 보인다
- 스크롤 흐름이 자연스러워야 한다
- 텍스트는 쉬운 표현 위주로 쓴다
- 긴 설명은 접기/펼치기 처리한다

## 모바일 필수 UX
- 지역 선택이 빠르고 쉬워야 한다
- 지도 2D/3D 전환이 쉬워야 한다
- 차트는 작은 화면에서도 읽혀야 한다
- 전화/링크 버튼은 즉시 눌러야 한다
- 지도 없이도 기관 목록을 리스트로 볼 수 있어야 한다

# 12. 시각화 규칙

## 공통 규칙
모든 시각화는 다음을 충족해야 한다.
- 제목
- 단위
- 기준연도
- 출처
- 해석 문구
- 데이터 한계 설명

## 해석 문구 유형
시각화마다 최소 한 가지 이상 포함한다.

### 긍정 해석
- “값이 높을수록 상대적으로 안전망이 잘 갖춰진 것으로 볼 수 있습니다.”

### 부정 해석
- “값이 높을수록 주의가 필요한 신호일 수 있습니다.”

### 주의 해석
- “지역 비교를 위한 참고 자료이며 단순 인과관계를 의미하지 않습니다.”

### 한계 설명
- “조사 방식, 표본 특성, 기준연도 차이를 고려해 해석해야 합니다.”

## 지도 시각화 규칙
- 2D/3D 전환 가능
- 기관 유형별 구분 가능
- 위험 표현은 과하지 않게
- 베타 정보는 명확히 표시
- 색상 외 텍스트 설명 제공
- 클릭 시 상세 정보 카드 제공

# 16. 오류/빈 데이터 UX 규칙

이 항목은 반드시 구현 기준에 포함한다.

## 빈 데이터일 때
- “해당 지역의 최신 데이터가 아직 준비되지 않았습니다.”
- “조건에 맞는 기관 정보가 없습니다.”
- “선택한 연도의 비교 가능한 자료가 없습니다.”

## 오류가 발생했을 때

- 이해 가능한 안내 문구를 제공할 것
- 재시도 또는 다른 경로 안내를 제공할 것

## 부분 데이터만 있을 때
- 표시 가능한 데이터는 보여주고
- 부족한 데이터는 “준비 중” 또는 “일부만 제공”으로 명시

# 22. 절대 놓치면 안 되는 핵심 요구사항

1. 청소년, 학부모, 선생님 모두가 이해할 수 있어야 한다  
2. 모바일 우선으로 설계해야 한다  
3. 지도는 2D와 3D 모두 가능해야 한다  
4. 모든 시각화에는 해석이 있어야 한다  
5. 긍정/부정/주의 방향이 명확해야 한다  
6. 도움 연결 기능이 반드시 있어야 한다  
7. API 키를 활용한 자동 점검/업데이트 구조가 필요하다  
8. 기존 Streamlit 결과물과 파일을 최대한 재사용해야 한다  
9. 지역별 맞춤 정보 제공이 핵심이다  
10. 출처와 데이터 설명이 반드시 있어야 한다  
11. 운영자 관점의 점검 구조도 고려해야 한다  
12. 빈 데이터/오류 상황 UX를 반드시 정의해야 한다  
13. 서비스 성과를 측정할 KPI를 포함해야 한다
